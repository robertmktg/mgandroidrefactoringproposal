$ Android Architecture
Refactoring Roadmap

A structured proposal for transitioning the marktguru Android app from its current monolithic codebase to a modern, modular, and fully testable architecture — enabling long-term speed, quality, and team scalability.

Why This Matters Now
Our Android codebase has reached a critical threshold
Every new feature we add to the current architecture costs more time than the previous one. Without action, development velocity will continue to decline while bugs and fragility increase.
⚡

The core problem in one sentence

We have a monolithic app written in mixed Java & Kotlin, with XML UI tightly coupled to business logic through Presenters — making it nearly impossible to test, scale, or safely refactor individual features.

Current State
Six critical problems with today's codebase
These are not theoretical concerns — they are actively slowing down every sprint today.
🏗️

Monolithic structure

All features share the same code space. A change in one area risks breaking unrelated parts. There are no clear boundaries between features, making navigation of the code difficult and refactoring dangerous.

Highest Impact
🔗

Presenters tightly coupled to XML UI

Business logic lives inside or directly next to UI code (XML layouts + Presenter classes). The UI framework — which is already in maintenance mode — is woven through the logic layer. Changing one requires changing the other.

Blocks UI Migration
🧪

Virtually untestable codebase

Because Presenters are bound to the Android framework and XML Views, unit tests cannot run independently of a device or emulator. There is no reliable way to write fast, isolated tests for business logic — meaning bugs reach production.

No Safety Net
☕

Java classes in a Kotlin-first world

A significant portion of the codebase is still in Java. This prevents use of modern Kotlin features like coroutines, sealed classes, and value classes. Java/Kotlin interop creates hidden complexity and friction for every developer on the team.

Technical Debt
📱

XML UI is in maintenance mode

Google no longer actively invests in the XML View system. Jetpack Compose is the present and future of Android UI. Staying on XML means missing out on modern tooling, better performance, and simpler state handling — and it will become increasingly hard to hire for.

Platform Risk
🐌

Feature velocity will keep decreasing

Every new feature added to a monolith increases the weight of everything else. Compile times grow. Side effects multiply. Code reviews become harder. What takes 2 days today will take 5 days in 12 months if we do not change the architecture now.

The golden rule

Features never depend on each other. If two features need the same data model, it moves to :core:domain. If they share a UI component, it moves to :core:design-system. This boundary is enforced at the Gradle level — a build error prevents accidental cross-feature dependencies before they ever reach code review.

Dependency Injection
Koin vs Hilt — choosing the right DI framework
The new architecture requires a proper DI framework to wire modules together cleanly. We will evaluate both leading options and choose the best fit for our project before Phase 1 begins.
KOIN
insert-koin-android-ext
// Pure Kotlin, zero annotation processing
✓ Pure Kotlin DSL — no annotation processors, no kapt/ksp required for DI itself
✓ Faster build times — no code generation step
✓ Simpler setup in multi-module projects — modules just declare their own Koin modules
✓ Great fit for Kotlin Multiplatform (KMP) if we ever go cross-platform
✗ Runtime DI resolution — errors surface at runtime, not compile time
✗ Less first-party Google tooling integration (WorkManager, Navigation, etc.)
HILT
com.google.dagger:hilt-android
// Google-recommended, compile-time safe
✓ Compile-time verification — DI errors caught at build time, not production
✓ First-party Google support — deep integration with Jetpack (Navigation, WorkManager, Compose)
✓ Industry standard — easier to hire developers already familiar with it
✗ Requires ksp annotation processing — adds build time overhead
✗ More boilerplate in multi-module setups (component dependencies, subcomponents)
✗ Android-only — not usable if we ever target KMP/non-Android platforms
🔍

Our approach: evaluate and decide in Phase 1

Both frameworks are production-proven on large Android apps. The right choice depends on factors specific to our team: existing familiarity, future KMP ambitions, and preferred trade-off between build speed (Koin) and compile-time safety (Hilt). We will run a structured spike in Phase 1 — build the same :core:di module with both, benchmark build time impact, and choose the one that best fits our long-term roadmap. The rest of the architecture is framework-agnostic and is not affected by this decision.

Important Note for Management
We cannot "just switch to Compose"
This is a common misconception that needs to be addressed directly before planning begins.
🚧

Compose requires ViewModel — Presenter is not compatible

Jetpack Compose is built around state flowing down from ViewModels. Our current Presenters hold a direct reference to the View (XML layout), which fundamentally conflicts with Compose's unidirectional data flow model. Migrating to Compose without first replacing Presenters with ViewModels would produce a system that is worse than either approach alone.

1️⃣

First: Architecture layer

Replace Presenters with ViewModels using MVI (State + Actions + Events). This decouples business logic from any UI framework, making it testable and Compose-ready.

2️⃣

Then: Kotlin migration

Replace all remaining Java classes with Kotlin. This unlocks coroutines, sealed classes, and eliminates the interop overhead that slows every developer down.

3️⃣

Finally: Compose UI

Once ViewModels are in place and the architecture is clean, Compose migration becomes incremental and safe — screen by screen, feature by feature.

Testability
Today's code cannot be tested safely
This is not a quality preference — it is a structural impossibility that directly causes production bugs.

❌ Current state — no testability

Presenters hold a reference to the Android View. Running a test requires spinning up an Activity or Fragment, which means tests only run on a device or emulator — slowly. Business logic cannot be verified in isolation. A change in one part of the UI can silently break unrelated logic with no automated check to catch it.

✅ Target state — full testability

ViewModels expose state as a Flow. Business logic in use cases and repositories depends only on interfaces. Tests run in milliseconds on the JVM with no device needed. Every feature's ViewModel, repository, and use case can be fully covered with fast, reliable unit tests using JUnit5 + Turbine + fake repositories.

Migration Roadmap
Four phases — incremental, low-risk delivery
The app is never rewritten from scratch. Each phase delivers working, shippable software. Old and new code coexist during the transition.
1

Foundation — Build the new skeleton

Estimated: 6–8 weeks

Set up the multi-module Gradle structure with convention plugins, version catalogs, and the full :core module hierarchy. No user-visible change. This phase is purely enabling infrastructure — a new home where all future work will live. Also establish the testing infrastructure (JUnit5, Turbine, fake repositories) so every new module is born testable. Critically, this phase includes a structured spike to evaluate Koin vs Hilt and select the DI framework for the project (see dedicated section below).

Gradle multi-module setup Convention plugins :core modules Version catalog Test infrastructure DI framework spike (Koin vs Hilt)
2

Java → Kotlin, Presenter → ViewModel

Estimated: 3–5 months (feature by feature)

For each feature, in order of priority: convert remaining Java classes to Kotlin, extract business logic from Presenters into ViewModels with MVI state, write unit tests for the new logic. The XML UI stays intact during this phase — ViewModels work with both XML and Compose, so this migration is safe to do before touching the UI layer.

Java → Kotlin Presenter → ViewModel MVI state design Unit test coverage Feature isolation
3

XML → Jetpack Compose

Estimated: 3–6 months (screen by screen)

With ViewModels in place, the UI migration to Compose becomes safe and incremental. Each screen is migrated individually. Compose interop allows old XML and new Compose screens to coexist in the same app at the same time. The :core:design-system module is built here, establishing the shared Compose component library for all features.

Jetpack Compose Design system Compose UI tests Compose Navigation Screen-by-screen
4

Full modularization & cleanup

Estimated: 4–8 weeks

Complete the feature module extraction. Remove all legacy XML layouts and Presenter classes. Enforce Gradle dependency rules that prevent new cross-feature coupling. At this point the codebase matches the target architecture fully — new features can be added as self-contained modules without touching anything that exists.

Full feature modules Remove legacy code Enforced boundaries API/Impl modules Architecture docs
Risk Assessment
The cost of doing nothing
Delaying this refactoring is not a neutral choice — it actively increases risk and cost over time.
Critical

Feature delivery slows to a crawl

As the monolith grows, each new feature added increases the complexity of the entire system. What takes 2 developer-days today will take 4–5 days in 12 months. Product commitments become unreliable.

Critical

Production bugs with no safety net

Without testable architecture, every change is manual-QA-only. As the codebase grows, edge cases multiply and manual testing cannot cover them. The current setup guarantees an increasing rate of production regressions.

High

Talent acquisition becomes harder

Senior Android engineers expect modern tooling: Kotlin, Compose, MVI, clean architecture. Advertising a position that uses Java, XML, and Presenters as primary tools will narrow the talent pool significantly and signal technical debt to candidates.

High

XML View system will lose support

Google's investment in the XML View system has effectively stopped. Compose is the strategic direction for Android UI. The longer we stay on XML, the further we fall behind on platform capabilities, performance improvements, and OS-level UI features.

Medium

Parallel feature development becomes impossible

In the current monolith, two developers working on different features frequently conflict in version control. Modular architecture allows completely parallel, conflict-free development — a prerequisite for scaling the team.

Expected Outcomes
What the team gains
🚀

Faster feature delivery

Isolated modules mean new features never risk breaking existing ones. Developers work with confidence and fewer surprises.

🧪

Reliable automated testing

ViewModels and repositories are testable in milliseconds. Regressions are caught before review, not after deployment.

⚡

Faster build times

Gradle only recompiles modules that changed. In a full multi-module project, incremental builds can be 40–60% faster.

🤝

Team scalability

Multiple engineers can own different feature modules without stepping on each other. Clear ownership, fewer merge conflicts.

🎨

Modern UI with Compose

Jetpack Compose unlocks faster UI iteration, animated components, and full access to the latest Material Design 3 tooling.

📖

Easier onboarding

A modular codebase with clear conventions means new team members are productive in days, not weeks.

🎯

The right time to start was a year ago. The second best time is now.

Every sprint we add new features to the current monolith is a sprint that makes the eventual migration larger and more expensive. This is not a request to stop feature development — it is a request to invest in the foundation that makes all future feature development faster, safer, and more sustainable. The phases are designed to deliver alongside product work, not instead of it.
